## Proyecto

El proyecto **Sistema Web orientado a la Gestión Inteligente de la Farmacia Vida Sana** tiene como objetivo desarrollar un Sistema Web que facilite la gestión de ventas, inventario y clientes en dicha farmacia, optimizando sus procesos operativos y mejorando la experiencia del usuario.

- **Comunicación:** Utilizamos Discord para mantenernos actualizados y comunicarnos eficazmente, con un canal dedicado para dudas rápidas y otro para anuncios importantes.

## Base de Datos

Una base de datos es un sistema organizado diseñado para almacenar, gestionar y recuperar información de manera eficiente y estructurada. En el contexto del Sistema Web para una Farmacia, la base de datos actúa como el núcleo del sistema, permitiendo la administración de datos críticos como la información de clientes, el inventario de productos (medicamentos y artículos), los pedidos realizados, los detalles asociados a cada transacción, las ventas y la gestión de usuarios y cajas. Este componente asegura que la información esté disponible de forma rápida y segura para los usuarios del sistema, como empleados y administradores, optimizando procesos como la venta, el control de stock y la atención al cliente.

Para este proyecto, se ha diseñado una base de datos relacional que utiliza tablas interconectadas mediante claves primarias y foráneas para garantizar la integridad y consistencia de los datos. Las principales entidades identificadas incluyen "Caja", "Categoría", "Cliente", "Usuario", "Producto", "Venta", "Detalle_Pedido", "Venta_Detalle", "Empresa" y "Pedido", cada una con atributos específicos que responden a las necesidades operativas de la farmacia. Este enfoque relacional permite realizar consultas complejas, como el historial de compras de un cliente, el estado actual del inventario o el registro de ventas por caja, de manera eficiente, soportando así las funcionalidades clave del sistema web.

La implementación de la base de datos no solo facilita la gestión diaria de la farmacia, sino que también proporciona una base escalable para futuras expansiones, como la integración de módulos adicionales (por ejemplo, gestión de proveedores o análisis de ventas). En resumen, la base de datos es el fundamento que asegura la operatividad, confiabilidad y adaptabilidad del sistema, alineándose con los objetivos de mejorar la experiencia del usuario y la eficiencia operativa.

**Diagrama de Base de Datos**  
El diagrama de base de datos proporcionado es un Diagrama Entidad-Relación (ER) que representa el modelo lógico de una base de datos relacional diseñada para el Sistema Web de una Farmacia. Este diagrama incluye las entidades principales, sus atributos, y las relaciones entre ellas, detallando claves primarias, foráneas, tipos de datos y restricciones.

- **Elementos Indicados:**
  - **Llaves primarias (PK):** Se identifican con la notación **PK**. Cada tabla tiene su identificador único:
    - caja_id en la tabla Caja.
    - categoria_id en la tabla Categoria.
    - cliente_id en la tabla Cliente.
    - usuario_id en la tabla Usuario.
    - producto_id en la tabla Producto.
    - venta_id en la tabla Venta.
    - detalle_id en la tabla Detalle_Pedido.
    - venta_detalle_id en la tabla Venta_Detalle.
    - empresa_id en la tabla Empresa.
    - pedido_id en la tabla Pedido.
  - **Llaves foráneas (FK):** Se indican con la notación **FK**. Estas claves establecen relaciones entre tablas:
    - caja_id en la tabla Usuario, que hace referencia a Caja.
    - caja_id en la tabla Venta, que hace referencia a Caja.
    - categoria_id en la tabla Producto, que referencia a Categoría.
    - cliente_id en la tabla Venta, que referencia a Cliente.
    - usuario_id en la tabla Venta, que referencia a Usuario.
    - producto_id en la tabla Detalle_Pedido, que referencia a Producto.
    - pedido_id en la tabla Detalle_Pedido, que referencia a Pedido.
    - producto_id en la tabla Venta_Detalle, que referencia a Producto.
    - venta_id (implícito mediante venta_codigo) en la tabla Venta_Detalle, que referencia a Venta.
- **Tipos de datos y restricciones:** Cada atributo incluye su tipo de dato y restricciones implícitas:
  - **Tipos de datos utilizados:** int (e.g., caja_id, cliente_id), varchar (e.g., cliente_nombre, producto_codigo), decimal (e.g., venta_total, producto_precio_venta), date (e.g., venta_fecha, fecha en Pedido), y enum (e.g., estado en Pedido).
  - **Restricciones:** Según el diccionario de datos, todos los campos son requeridos, lo que implica que no se permiten valores nulos para ningún atributo.
- **Relaciones:**  
  Las asociaciones entre entidades representan las relaciones entre tablas:
  - **1 a 1:** No se observan relaciones de este tipo en el diagrama.
  - **1 a muchos:**
    - Un Cliente puede tener muchas Ventas (relación entre Cliente y Venta mediante cliente_id).
    - Una Categoría puede estar asociada a muchos Productos (relación mediante categoria_id).
    - Una Caja puede estar asociada a muchos Usuarios y Ventas (relaciones mediante caja_id).
    - Un Usuario puede registrar muchas Ventas (relación mediante usuario_id).
    - Un Pedido puede tener muchos Detalles_Pedido (relación mediante pedido_id).
  - **Muchos a muchos:** Se modela con tablas intermedias:
    - La relación entre Venta y Producto se gestiona mediante la tabla intermedia Venta_Detalle, que incluye venta_id (implícito mediante venta_codigo) y producto_id como claves foráneas.
    - La relación entre Pedido y Producto se gestiona mediante la tabla intermedia Detalle_Pedido, que incluye pedido_id y producto_id como claves foráneas.

Este diseño relacional asegura la integridad de los datos y permite consultas eficientes, como el historial de ventas por cliente, el detalle de productos en un pedido, o el control de stock por categoría, soportando las necesidades operativas de la farmacia.
